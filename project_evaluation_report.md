# 评估报告成文思路指令

## 0. 一句话定义

这是一套用于生成**领导汇报型可行性评估报告**的写作指令。

它的核心不是“把资料整理完整”，而是帮助领导快速判断：

> 这件事值不值得做、能不能做、当前能做到什么程度、风险在哪里、下一步应该怎么验证或落地。

报告必须做到：

* 结论先行；
* 主线清楚；
* 每节都有说明和判断；
* 证据服务结论；
* 风险和建议明确；
* 不把调研材料、工具测试、数据表格写成流水账。

---

# 一、适用场景

本指令适用于以下类型报告：

1. 新功能可行性评估；
2. 产品需求立项评审；
3. 技术落地可行性判断；
4. 第三方工具/平台是否可接入评估；
5. POC 是否值得做的判断报告；
6. 项目是否继续推进、暂缓、降级或转向的决策报告；
7. 面向领导、老板、客户、售前、项目评审会的汇报材料。

不适用于纯知识科普、纯竞品罗列、纯测试记录、纯技术文档。

---

# 二、总写作原则

## 2.1 报告主线必须是“决策”，不是“资料”

报告不能按“我查到了什么”来组织，而要按：

```text
这件事要不要做
→ 为什么值得/不值得做
→ 当前证据说明什么
→ 最大风险在哪里
→ 下一步怎么验证或落地
```

如果报告读完后，领导还不知道该不该做、下一步做什么，这份报告就是失败的。

---

## 2.2 每一节都必须有“说明 + 判断/结论”

每个一级章节、重要二级章节，都要遵循：

```text
说明：这一节在讲什么事实、背景或证据
判断：这些事实说明什么
结论：对项目有什么影响，下一步应该怎么做
```

不要只堆信息。

错误写法：

> 工具 A 支持 PDF，工具 B 支持 OCR，工具 C 支持表格识别。

正确写法：

> 当前工具调研说明，市面上已有 OCR、表格识别、文档翻译等可复用能力，因此项目并非从零建设；但这些能力尚未在客户真实样本上完成验证，因此当前只能支持进入 POC，不能支持直接完整开发。

---

## 2.3 证据必须服务结论

工具测试、样例文件、用户访谈、调研资料都不是主线。

它们的作用是支撑判断。

写作时要避免：

```text
第一，测试了 DeepL；
第二，测试了 Google；
第三，测试了 BluTranslate；
第四，测试了 PaddleOCR；
```

应该写成：

```text
现有翻译软件测试结果共同说明：普通文档翻译能力已经具备一定基础，但在正式交付所需的结构稳定性、页数完整性、图表文字、表格行列关系上仍存在明显风险。因此，本项目当前支持做 POC，但不支持直接承诺正式交付。
```

---

## 2.4 未验证内容必须明确标注

报告必须区分三类信息：

| 类型    | 写法                       |
| ----- | ------------------------ |
| 已验证事实 | 当前已通过样例/测试/数据验证，可以作为判断依据 |
| 合理推断  | 基于样例和调研得出的推断，需要说明前提      |
| 待验证事项 | 尚未实测，不能写成已确认结论           |

禁止把“官网说支持”“理论上可以”“看起来能做”写成“已经可用”。

正确写法：

> PaddleOCR 官方能力覆盖 OCR、表格识别和版面分析，但当前尚未用客户样本完成实测，因此只能作为 POC 候选组件，不能作为已验证技术结论。

---

## 2.5 报告要敢下判断，但不要乱下判断

好的报告不是模棱两可。

必须给出明确结论：

* 完全可行；
* 条件可行；
* 暂不可行；
* 建议进入 POC；
* 不建议直接开发；
* 建议降级为辅助工具；
* 建议暂缓。

但结论必须和证据对应，不能为了显得有判断而过度承诺。

---

# 三、报告整体结构

推荐使用以下结构。

```text
封面
一、评估概述
二、需求背景与业务边界
三、已验证事实与样例复杂度
四、现有方案/工具/测试结果评估
五、业务与立项可行性
六、技术可行性判断
七、风险评估与应对方案
八、POC / 落地方案
九、综合结论与决策建议
十、附录：证据材料
```

如果报告较短，可以合并章节，但不能丢失以下核心内容：

* 摘要；
* 最终结论；
* 已验证事实；
* 关键风险；
* 技术可行性判断；
* POC 或落地建议；
* 综合决策建议。

---

# 四、封面怎么写

封面要简洁、高级、管理层可读。

## 4.1 封面必须包含

| 内容   | 写法                         |
| ---- | -------------------------- |
| 报告标题 | 项目名称 + 可行性评估报告             |
| 副标题  | 基于什么材料/什么阶段的判断             |
| 版本   | V1.0 / V2.0 / 最终版 / 阶段性评估版 |
| 日期   | 报告日期                       |
| 评估对象 | 评估的功能、项目、工具或方案             |
| 最终结论 | 一句话写清楚                     |
| 报告边界 | 当前评估什么，不评估什么               |

## 4.2 标题写法

推荐：

```text
PDF 专业翻译能力初步可行性评估报告
```

或：

```text
XX 功能可行性评估报告
```

副标题示例：

```text
基于客户需求、样例文件与多工具测试结果的 POC 决策建议
```

## 4.3 封面摘要卡片

封面或第一页可以放一个摘要表：

| 项目    | 内容                     |
| ----- | ---------------------- |
| 评估对象  | XX 功能                  |
| 评估目的  | 判断是否值得进入 POC / 立项 / 开发 |
| 当前结论  | 条件可行，建议先 POC           |
| 核心风险  | OCR、表格结构、版式回填          |
| 下一步建议 | 2 周 POC 验证             |

封面不要放大量细节，不要出现过多工具名称。

---

# 五、开头摘要怎么写

摘要是整份报告最重要的部分。

领导可能只看摘要，所以摘要必须回答 5 个问题：

1. 这次评估什么？
2. 当前结论是什么？
3. 为什么得出这个结论？
4. 关键风险是什么？
5. 下一步建议是什么？

## 5.1 摘要结构

推荐写成：

```text
本次评估对象是【功能/项目】。
当前结论是：【完全可行 / 条件可行 / 暂不可行】。
该结论基于三类证据：【客户需求】、【样例/测试结果】、【技术调研】。
现有证据说明：【可行点】；但同时暴露出【核心风险】。
因此，建议：【进入 POC / 直接立项 / 暂缓 / 降级】。
```

## 5.2 摘要示例

```text
本次评估对象是 PDF 专业翻译能力，目标是判断该能力是否支持进入 POC 及后续产品化建设。

当前结论是：条件可行，建议进入 POC，但不建议直接完整开发或承诺一键生成正式终稿。

现有多款翻译软件测试结果说明，工具在规则化表格、部分专业术语、视觉版式保留方面已经具备一定基础，因此项目不是完全不可做；但测试也暴露出页数不完整、表格行列关系不稳定、图表标题和坐标轴漏译、表单控件错位、编号和 Program 被改写等 P0 风险。

因此，下一步不应继续停留在方案层，也不应直接进入完整开发，而应先做 2 周 POC，重点验证页数完整性、结构化识别、表格对齐、图表文字、不可翻译字段保护和输出可接受度。
```

---

# 六、每个章节怎么写

## 6.1 每章固定写法

每个章节建议使用以下结构：

```text
章节开头：说明这一章解决什么问题
正文：列出事实、证据、观察
判断：说明这些事实意味着什么
本节结论：对项目的影响
```

## 6.2 每节结尾必须有判断

每个重要章节最后加一句：

```text
本节判断：
……
```

或：

```text
对项目的影响：
……
```

或：

```text
因此，本节结论是：
……
```

这样报告读起来会非常清楚。

---

# 七、各章节具体写法

## 一、评估概述

### 作用

让领导快速知道这件事到底该怎么看。

### 必须包含

* 评估对象；
* 评估目的；
* 当前阶段；
* 最终结论；
* 关键依据；
* 下一步建议。

### 推荐结构

```text
1.1 一句话结论
1.2 领导决策摘要
1.3 当前判断边界
```

### 一句话结论写法

```text
当前项目具备继续推进价值，但只能支持进入 POC，不支持直接完整开发。
```

或：

```text
该功能具备落地基础，但必须满足 XX 条件后再启动。
```

### 表格放置

评估概述适合放“领导决策摘要表”。

示例：

| 决策项      | 建议结论      | 依据             |
| -------- | --------- | -------------- |
| 是否值得继续   | 值得        | 客户需求明确，价值真实    |
| 是否直接开发   | 不建议       | 关键技术风险未验证      |
| 是否进入 POC | 建议        | POC 可低成本验证成败关键 |
| 最大风险     | 表格结构和版式回填 | 已在样例中暴露问题      |

---

## 二、需求背景与业务边界

### 作用

解释为什么要做这件事，以及客户到底要什么。

### 必须包含

* 业务现状；
* 用户诉求；
* 功能定义；
* 输入输出；
* 当前边界；
* 不确定事项。

### 写法

先说明客户需求，再判断需求是否真实。

示例：

```text
客户当前希望在现有平台中增加文档翻译能力，用户上传 PDF 后系统自动输出翻译后的 PDF，并可叠加 Word 和双语对照稿。该需求不是单纯的阅读辅助，而是面向批量业务资料处理的效率型能力。

本节判断：需求具备真实性和业务价值，但客户对“一键生成可用终稿”的预期偏理想化，必须在 POC 阶段进一步确认是否接受人工审校和多种输出形态。
```

### 表格放置

需求背景适合放“客户需求拆解表”。

| 需求项  | 当前信息          | 对项目的影响      |
| ---- | ------------- | ----------- |
| 输入文件 | PDF 为主        | PDF 解析是核心风险 |
| 输出要求 | PDF、Word、双语对照 | 需要导出和审校能力   |
| 客户期望 | 一键生成可用结果      | 需要校准预期      |
| 人工审校 | 未确认           | 是项目边界关键     |

---

## 三、已验证事实与样例复杂度

### 作用

说明现在手里有哪些事实，不是凭空判断。

### 必须包含

* 已有样例；
* 已做测试；
* 文件类型；
* 文件复杂度；
* 复杂度对项目的影响。

### 写法

不要直接罗列文件名，而要按复杂度归类。

示例：

```text
从样例文件看，项目处理对象并不是普通正文 PDF，而是包含表单、图表、横向宽表、多级表头和大量数据字段的复杂 PDF。不同文件类型对系统能力的要求不同，不能用单一“PDF 翻译”逻辑覆盖。

本节判断：样例文件复杂度足以说明，该项目的技术核心不是翻译模型，而是 OCR、版面解析、表格结构、图表文字和版式回填。
```

### 表格放置

样例分析适合放“文件类型复杂度表”。

| 文件类型 | 样例特征        | 主要难点     | 风险等级 |
| ---- | ----------- | -------- | ---- |
| 表单类  | 勾选框、横线、固定字段 | OCR、版式回填 | 高    |
| 图表类  | 坐标轴、图例、单位   | 图表文字识别   | 高    |
| 宽表类  | 多级表头、数据密集   | 行列关系     | 极高   |
| 长表类  | 表头重复、页数大    | 批量稳定性    | 中高   |

---

## 四、现有方案 / 工具 / 测试结果评估

### 作用

用工具测试结果支撑判断，但不能让工具测试抢主线。

### 必须包含

* 已测试工具；
* 工具表现；
* 暴露问题；
* 对项目判断的影响。

### 写法

先给总判断，再给工具对比。

示例：

```text
现有翻译软件测试结果共同说明，当前工具已具备一定初翻能力，尤其在规则化表格和部分专业术语上有可利用价值。但这些工具在正式交付所需的页数完整性、表格结构、图表文字、表单版式和不可翻译字段保护上仍不稳定。

本节判断：工具测试结果支持继续做 POC，但不支持直接完整开发或承诺正式交付。
```

### 表格放置

工具评估表要在这段判断之后放。

| 工具   | 相对优势   | 暴露问题    | 对 POC 的启示 |
| ---- | ------ | ------- | --------- |
| 工具 A | 初翻能力较好 | 版式差     | 只能做基线     |
| 工具 B | 术语更好   | 丢页、图表漏译 | 值得进入 POC  |
| 工具 C | 结构化能力强 | 未实测     | 候选组件      |

### 禁止写法

不要写成：

```text
4.1 DeepL
4.2 Google
4.3 BluTranslate
4.4 PaddleOCR
```

除非这是工具选型报告。

如果是可行性评估报告，工具只是证据，应写成：

```text
4.1 通用翻译工具暴露的问题
4.2 专业翻译工具暴露的问题
4.3 对 POC 的启示
```

---

## 五、业务与立项可行性

### 作用

判断这是不是值得做的项目，不是伪需求。

### 必须包含

* 业务刚需性；
* 行业落地性；
* 项目价值；
* 公司战略或产品能力匹配；
* 立项必要性结论。

### 写法

示例：

```text
从业务侧看，该需求具备明确的页量规模、重复处理场景和交付效率诉求，不属于单点体验优化。尤其在大批量文档、固定模板、重复表头和术语一致性场景下，平台化能力具备降本增效价值。

本节判断：项目具备立项调研价值，但当前不应以完整开发立项，而应以 POC 立项。
```

### 表格放置

| 维度   | 判断     | 依据         |
| ---- | ------ | ---------- |
| 业务刚需 | 高      | 客户有明确处理场景  |
| 价值空间 | 中高     | 可降本增效      |
| 行业壁垒 | 中      | 需要专业文档处理能力 |
| 立项建议 | POC 立项 | 技术风险未闭环    |

---

## 六、技术可行性判断

### 作用

判断技术上能不能做，难点在哪里，当前证据支持到什么程度。

### 必须包含

* 技术链路；
* 已有能力；
* 技术难点；
* 未验证事项；
* 技术结论。

### 写法

示例：

```text
从技术链路看，该项目需要完成“文档上传 → OCR/版面解析 → 表格/图表识别 → 翻译 → 术语约束 → 版式回填 → 质检 → 导出”的完整闭环。当前已有翻译软件测试证明初翻能力存在基础，但 OCR、结构化识别、版式回填和批量稳定性仍未被真实样本验证。

本节判断：技术方向有限可行，但当前仅支持进入 POC，不支持直接判断可完整交付。
```

### 表格放置

技术章节适合放“模块可行性表”。

| 模块   | 当前判断 | 证据      | 是否需 POC |
| ---- | ---- | ------- | ------- |
| 文档上传 | 可行   | 常规平台能力  | 否       |
| OCR  | 未验证  | 仅官方文档   | 是       |
| 表格结构 | 高风险  | 样例暴露问题  | 是       |
| 图表文字 | 高风险  | 漏译/误识别  | 是       |
| 翻译引擎 | 部分可行 | 多工具初翻可用 | 是       |
| 导出   | 高风险  | 版式错位    | 是       |

---

## 七、风险评估与应对方案

### 作用

让领导知道项目失败点在哪里，以及如何控制。

### 必须包含

* 风险点；
* 风险等级；
* 事实依据；
* 影响；
* 应对方案；
* 是否是 POC 必验项。

### 写法

示例：

```text
当前最大风险不是翻译结果是否自然，而是结果是否完整、结构是否可信、关键字段是否被误改。这些风险一旦进入正式交付，会直接影响客户验收和业务可信度。

本节判断：风险集中在 P0 技术验证项上，必须通过 POC 前置验证，不能在完整开发阶段再暴露。
```

### 表格放置

风险章节必须用表格。

| 优先级 | 风险点   | 事实依据        | 影响 | 应对方案         |
| --- | ----- | ----------- | -- | ------------ |
| P0  | 页数丢失  | 译文页数少于原文    | 致命 | 页数一致性校验      |
| P0  | 表格错位  | 行列关系不稳定     | 致命 | 结构化识别 + 抽样核对 |
| P0  | 字段被改  | 编号/日期/ID 改写 | 高  | 不可翻译规则       |
| P1  | 术语不一致 | 工具翻译不统一     | 中高 | 术语库          |

---

## 八、POC / 落地方案

### 作用

说明下一步怎么验证，验证到什么程度算通过。

### 必须包含

* POC 目标；
* 样本范围；
* 验证项；
* 验收标准；
* 通过后的动作；
* 不通过后的动作。

### 写法

示例：

```text
POC 的目标不是证明系统能翻译一句话，而是验证客户真实文件能否稳定走通“识别、翻译、审校、导出”链路。POC 应重点验证页数完整性、表格行列关系、图表文字、不可翻译字段保护和输出格式可接受度。

本节判断：POC 是当前最合理的下一步。若 POC 通过，可进入 MVP；若 POC 失败，应将项目降级为辅助翻译/审校工具。
```

### 表格放置

POC 章节必须放“验证计划表”。

| 验证项   | 样本范围         | 通过标准        | 不通过时判断  |
| ----- | ------------ | ----------- | ------- |
| 页数完整性 | 每类样本 10-50 页 | 输出页数=输入页数   | 不进入正式开发 |
| 表格结构  | 宽表/listing   | 行列对应≥99%    | 降级为辅助工具 |
| 图表文字  | 统计图表         | 图题/坐标轴/图例完整 | 改为人工审校  |
| 字段保护  | 日期/ID/编号     | 0 改写        | 先建规则引擎  |
| 输出格式  | PDF/Word/双语稿 | 至少一种可接受     | 调整交付形态  |

---

## 九、综合结论与决策建议

### 作用

把整份报告收束到明确决策。

### 必须包含

* 最终结论；
* 是否建议做；
* 是否建议直接开发；
* 是否建议 POC；
* 下一步决策事项；
* 项目边界。

### 写法

示例：

```text
综合判断，本项目属于条件可行。现有证据说明，项目具备继续推进价值，但不具备直接完整开发条件。建议先做 2 周 POC，验证真实样本下的识别、结构、翻译、回填和导出质量。

若 POC 通过，可进入 MVP；若 POC 仅能完成文字初译但无法保证结构稳定，则项目应降级为辅助翻译/审校工具，不应承诺正式交付平台。
```

### 表格放置

最终结论适合放“决策建议表”。

| 问题       | 结论             |
| -------- | -------------- |
| 是否值得继续   | 值得             |
| 是否直接开发   | 不建议            |
| 是否进入 POC | 建议             |
| POC 目标   | 验证结构化识别和输出可接受度 |
| 客户承诺边界   | 不承诺无人审校终稿      |

---

# 八、表格使用规则

## 8.1 什么时候用表格

表格适合用于：

1. 对比多个对象；
2. 总结风险；
3. 拆解需求；
4. 呈现 POC 验收标准；
5. 展示决策项；
6. 汇总样例复杂度；
7. 归纳工具表现。

## 8.2 什么时候不要用表格

不要用表格替代判断。

如果一个内容需要解释因果关系，先写段落，再放表格。

错误写法：

> 直接放一张很大的工具功能矩阵。

正确写法：

```text
当前工具调研说明，市面上已有部分能力可复用，但没有单一工具能直接覆盖全部场景。因此工具矩阵的作用不是选定最终工具，而是识别哪些能力适合进入 POC。
```

然后再放表格。

## 8.3 表格前后必须有文字

每个重要表格前要有说明，表格后要有结论。

结构：

```text
先说明为什么放这张表
→ 放表格
→ 总结表格说明了什么
```

不要让表格孤零零出现。

---

# 九、段落写作规则

## 9.1 每段都要有作用

每个段落只承担一个作用：

* 说明背景；
* 陈述事实；
* 提炼问题；
* 做出判断；
* 给出建议。

不要一个段落同时塞太多信息。

## 9.2 推荐段落结构

```text
第一句：说明事实或背景。
第二句：解释这个事实为什么重要。
第三句：给出判断或对项目的影响。
```

示例：

```text
BluTranslate 在规则化长表格上的表现优于通用翻译工具，表头、部分术语和视觉版式有一定保留。这个结果说明专业翻译软件在部分标准化文档上具备可利用能力，因此项目并非完全不可做。但它同时出现疑似丢页、编号误识别和 Program 改写问题，因此当前结论只能是支持 POC，而不是支持直接开发。
```

## 9.3 每节结尾要收束

每节最后加：

```text
本节判断：
……
```

或：

```text
因此，本节结论是：
……
```

或：

```text
对项目的影响：
……
```

---

# 十、语言风格要求

## 10.1 整体风格

* 克制；
* 清楚；
* 直接；
* 敢下判断；
* 不夸张；
* 不写废话；
* 不写成 AI 味很重的空话。

## 10.2 推荐句式

```text
当前结论不是……，而是……
该结果说明……
这意味着……
因此，本阶段更合理的判断是……
现有证据可以支持……，但不能支持……
建议下一步……
该风险不应后置到开发阶段……
```

## 10.3 禁止句式

避免：

```text
随着技术不断发展……
在数字化时代背景下……
具有重要意义……
有望提升用户体验……
总体来看效果不错……
可以进一步优化……
```

这些话太空，不适合领导评估报告。

# 十一、证据使用规则

## 11.1 证据分类

报告中要明确区分：

```text
已验证事实：
已经通过样例、测试、用户反馈得到证实。

合理推断：
基于事实做出的推断，但仍需验证。

待验证事项：
目前尚未实测，不能作为结论依据。
```

## 11.2 证据表达方式

示例：

```text
已验证事实：Google 翻译后的长表样本视觉版式基本保留，但文本层顺序混乱，行列关系不适合作为数据级交付依据。

合理推断：规则化长表具备自动化潜力，但需要通过 POC 验证字段级对齐和页数完整性。

待验证事项：OCR 工具对 CRF 表单、统计图表和宽表 listing 的真实识别效果尚未实测。
```

---

# 十二、结论类型规则

最终结论只能选以下几类。

## 12.1 完全可行

适用条件：

* 需求明确；
* 技术成熟；
* 风险可控；
* 成本可接受；
* 已有足够验证结果。

写法：

```text
项目完全可行，建议进入开发排期。
```

## 12.2 条件可行

适用条件：

* 方向正确；
* 有业务价值；
* 部分能力已验证；
* 但仍存在关键风险；
* 需要 POC 或前置条件。

写法：

```text
项目条件可行，建议先做 POC，满足 XX 条件后再进入 MVP。
```

## 12.3 暂不可行

适用条件：

* 核心技术未突破；
* 成本明显过高；
* 客户目标不现实；
* 风险不可控；
* 缺少业务必要性。

写法：

```text
项目当前暂不可行，建议暂缓立项，或将目标降级为辅助工具。
```

# 十三、POC 类报告特殊规则

如果报告目标是判断是否做 POC，不能写成正式立项报告。

必须明确回答：

1. 为什么现在支持做 POC？
2. 为什么现在不支持直接开发？
3. POC 要验证什么？
4. POC 通过以后做什么？
5. POC 不通过以后怎么办？

POC 报告的核心句式：

```text
现有证据支持进入 POC，但不支持直接完整开发。
```

或：

```text
POC 的目标不是验证功能是否理想，而是验证关键风险是否可控。
```

# 十四、排版规则

## 14.1 整体排版

报告应有：

* 封面；
* 摘要卡片；
* 决策摘要表；
* 分层标题；
* 风险矩阵；
* POC 验证表；
* 综合结论表；
* 附录证据。

## 14.2 视觉风格

推荐：

* 白底；
* 深蓝 / 深灰作为主色；
* 风险用红/橙；
* 通过/建议用绿；
* 表格尽量简洁；
* 每页不要塞满；
* 标题层级明显；
* 页眉页脚统一。

## 14.3 表格风格

* 表头深色；
* 内容区浅色分隔；
* 重点风险可加 P0 / P1；
* 不要一页放过多列；
* 长句放段落，不塞表格。

# 十七、最终提示词模板

以后可以直接使用下面这段提示词。

```text
请帮我生成一份领导汇报型可行性评估报告。

报告目标：
判断【项目/功能】是否值得进入【POC/立项/开发】，并明确风险和下一步建议。

写作要求：
1. 主线必须是“是否值得做、能不能做、风险在哪里、下一步怎么验证”，不要写成资料汇总。
2. 结构采用：封面、评估概述、需求背景、已验证事实、测试/调研结果评估、业务与立项可行性、技术可行性、风险与应对、POC/落地方案、综合结论、附录。
3. 开头必须有摘要和一句话最终结论。
4. 每个章节必须有“说明 + 判断/结论”，不要只堆事实。
5. 表格只能用于支撑判断，不要为了堆信息而放表格。
6. 工具测试、样例问题、调研材料只能作为证据，不能抢主线。
7. 未实测内容必须明确写“待验证”，不能写成已验证。
8. 结论必须在“完全可行 / 条件可行 / 暂不可行”中选择一种。
9. 如果建议 POC，必须写清楚 POC 目标、样本范围、验证项、通过标准和失败后的处理方式。
10. 语言风格要像给领导看的评估材料：克制、清楚、敢下判断、每节能落到项目影响。
11. 排版要高级、整洁，适合直接汇报。
12. 参考我喜欢的写法：结论先行，每节有说明和判断，证据服务结论，不写工具测试流水账。
```
