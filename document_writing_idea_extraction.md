# 文档写作思路快速提炼指令

## 0. 一句话定义

这是一套用于分析任意优秀文档的**写作思路、结构逻辑、思考方式和可复用提示词**的指令。

它的目标不是总结文档内容，而是提炼：

> 这份文档为什么这样写、它的主线怎么组织、它的章节怎么展开、它的段落怎么收束、它的表格和图示什么时候出现、它的风格如何复刻。

---

# 一、使用场景

当用户上传一个文件，并希望学习它的写作方法时，使用本指令。

适用文件包括：

* 调研报告；
* 可行性评估报告；
* 售前方案；
* 产品方案；
* 行业研究；
* 竞品分析；
* 技术白皮书；
* 项目复盘；
* 汇报材料；
* PPT 文稿；
* 咨询报告；
* 领导汇报材料。

不适用于只想总结内容、提取数据、翻译文件的场景。

---

# 二、核心任务

模型拿到文件后，不要急着总结内容，而要站在“写作教练 + 产品经理 + 咨询顾问 + 结构分析师”的角度，回答以下问题：

1. 这份文档是什么类型？
2. 它写给谁看？
3. 它要支持什么决策？
4. 它的主线是什么？
5. 它的章节为什么这样排？
6. 它每一章承担什么作用？
7. 它每一节如何从事实走向判断？
8. 它的表格什么时候出现？
9. 它的图示/截图/案例什么时候出现？
10. 它的语言风格是什么？
11. 它有哪些可复用句式？
12. 如果我要写同类报告，应该如何下指令？

---

# 三、分析原则

## 3.1 不要只总结内容

错误方式：

```text
这份文档主要讲了 A、B、C。
```

正确方式：

```text
这份文档采用“问题提出 → 背景解释 → 结构拆解 → 案例验证 → 风险收束 → 落地建议”的写法。它不是按资料堆叠，而是先建立判断框架，再用材料支撑框架。
```

---

## 3.2 分析“怎么写”，不是“写了什么”

重点不是：

* 文档讲了什么内容；
* 里面有哪些知识点；
* 数据是多少；
* 案例有哪些。

重点是：

* 为什么先讲这个；
* 为什么后讲那个；
* 这个表格出现的作用是什么；
* 每个章节如何推进判断；
* 这套结构能不能复用到其他主题。

---

## 3.3 提炼“思考模型”

要把文档背后的思考方式提炼出来。

例如：

```text
这份报告的底层思路是：
先界定问题本质，再拆解行业演进，然后对比主流方案，最后提出推荐框架和落地路线。
```

或者：

```text
这份报告的思考方式不是“罗列竞品”，而是“从竞品中归纳行业范式，再反推自己的产品能力”。
```

---

## 3.4 提炼“成文指令”

分析完后，必须输出一段可以复用的提示词。

这段提示词要能让用户下次说：

> 按这个文档的写作思路，帮我写一份新的报告。

---

# 四、分析流程

## Step 1：先判断文档定位

拿到文件后，先判断：

| 分析项  | 要回答的问题                        |
| ---- | ----------------------------- |
| 文档类型 | 是调研报告、评估报告、方案、复盘，还是白皮书？       |
| 目标读者 | 是领导、客户、销售、产品、技术、投资人，还是内部团队？   |
| 文档目的 | 是为了决策、说服、科普、立项、交付，还是沉淀方法？     |
| 使用场景 | 是汇报、售前、立项、培训、评审，还是内部研究？       |
| 输出气质 | 是管理层风格、咨询风格、技术风格、产品风格，还是营销风格？ |

输出示例：

```text
这份文档属于内部深度研究报告，读者是产品、方案、技术和管理层。它的目的不是简单介绍概念，而是形成一套可用于后续方案设计和项目推进的分析框架。
```

---

## Step 2：提炼一句话主线

每份优秀文档都有一条主线。

要把它提炼成一句话。

常见主线类型：

| 主线类型  | 示例                  |
| ----- | ------------------- |
| 问题演进型 | 从旧问题到新问题，说明为什么需要新方案 |
| 决策判断型 | 判断这件事能不能做、值不值得做     |
| 架构推导型 | 从需求和风险出发，推导推荐架构     |
| 竞品归纳型 | 从多个案例中归纳行业范式        |
| 能力建设型 | 从当前能力缺口推导建设路线       |
| 风险治理型 | 从风险链路推导控制体系         |
| 方案落地型 | 从业务目标推导产品模块和实施路径    |

输出示例：

```text
这份报告的主线是：MaaS 平台从模型 API 走向 Agent 执行后，安全边界持续外扩，因此安全能力必须从接口防护升级为可信执行架构。
```

---

## Step 3：拆解整体结构

提取文档目录，并分析每个章节的作用。

不是简单复述目录，而是说明：

* 为什么这样排序；
* 每章承担什么功能；
* 章节之间如何递进；
* 哪一章是转折点；
* 哪一章是结论核心；
* 哪一章是落地部分。

输出格式：

| 章节   | 表面内容     | 实际作用     | 与前后文关系 |
| ---- | -------- | -------- | ------ |
| 执行摘要 | 核心判断     | 给领导快速结论  | 先定调    |
| 第一章  | 问题本质     | 解释为什么要研究 | 建立问题背景 |
| 第二章  | 政策标准     | 补充外部约束   | 增强可信度  |
| 第三章  | 架构范式     | 归纳行业做法   | 从资料到框架 |
| 第四章  | 平台案例     | 证明行业落地   | 支撑推荐方案 |
| 第五章  | 推荐架构     | 提出自己的框架  | 报告核心   |
| 后续章节 | 专题、风险、路线 | 落地展开     | 支撑执行   |

---

## Step 4：提炼章节写法

对每个章节分析：

1. 开头怎么写；
2. 先讲什么；
3. 再讲什么；
4. 是否有“本章结论”；
5. 表格什么时候出现；
6. 章节最后如何收束。

输出格式：

```text
第一章写法：
这一章不是直接解释概念，而是先讲行业/平台形态变化，再说明为什么旧安全边界失效，最后引出新概念和新风险。它的作用是建立“为什么必须研究这个问题”的必要性。
```

---

## Step 5：提炼段落写法

分析文档中典型段落的结构。

重点看：

* 是先结论还是先事实；
* 是否先说明背景，再给判断；
* 一段话一般多长；
* 判断句怎么写；
* 结尾是否收束；
* 有没有“因此”“这意味着”“本报告认为”等判断连接词。

输出格式：

```text
这份文档的段落通常采用：
事实说明 → 原因解释 → 判断提升 → 对后文的引导

例如：
先说明 MaaS 平台从模型 API 走向 Agent 执行；
再解释安全边界为什么外扩；
然后判断传统接口防护不够；
最后引出需要新的安全架构。
```

---

## Step 6：提炼表格使用规则

分析文档中表格出现的位置和作用。

不要只说“用了很多表格”，要说明表格为什么放在那里。

常见表格类型：

| 表格类型  | 使用场景  | 作用        |
| ----- | ----- | --------- |
| 阶段演进表 | 讲发展过程 | 让读者看到时间主线 |
| 概念对照表 | 讲多个概念 | 防止术语堆叠    |
| 政策标准表 | 讲合规要求 | 把政策转成落地含义 |
| 架构范式表 | 讲行业方案 | 归纳共性模式    |
| 平台案例表 | 讲竞品案例 | 对比可借鉴能力   |
| 风险控制表 | 讲风险治理 | 把问题转成措施   |
| 路线图表  | 讲落地建议 | 支撑项目推进    |

输出时要分析：

```text
这份文档的表格不是装饰，而是用于把复杂信息结构化。通常在一段判断之后出现，用来支撑刚刚的结论；表格之后又会继续收束，说明这个表格对后文有什么启示。
```

---

## Step 7：提炼图示/截图/案例使用规则

如果文档中有图示、截图、架构图、流程图，要分析：

* 图放在哪里；
* 图解决什么问题；
* 图前有没有说明；
* 图后有没有解读；
* 图是用来解释结构，还是证明案例；
* 图是否承担“框架化表达”的作用。

输出示例：

```text
这份文档中的架构图通常不是一开始就出现，而是在前面已经完成问题本质、行业范式和案例分析之后出现。这样读者看到图时，会理解它不是凭空设计，而是前文分析自然推导出的推荐框架。
```

---

## Step 8：提炼语言风格

分析文档语言风格。

需要判断：

| 维度   | 观察               |
| ---- | ---------------- |
| 句子长短 | 偏长句还是短句          |
| 判断力度 | 保守、中性、强判断        |
| 专业程度 | 小白友好、专业、专家级      |
| 术语密度 | 低、中、高            |
| 语气   | 汇报型、研究型、咨询型、产品型  |
| 连接词  | 因此、这意味着、本报告认为、建议 |
| 结论表达 | 是否每章有本章结论        |

输出示例：

```text
这份文档语言偏内部研究报告风格，判断明确但不夸张。它常用“本报告认为”“因此”“建议采用”“核心问题不是……而是……”这类句式，把资料上升为判断。
```

---

## Step 9：提炼可复用结构

把文档结构抽象成一个新主题也能套用的结构。

输出示例：

```text
如果复用这份文档的结构，可以抽象为：

1. 研究定位
2. 执行摘要
3. 问题本质
4. 行业演进
5. 政策标准
6. 主流范式
7. 平台案例
8. 推荐框架
9. 核心专题
10. 风险模型
11. 路线图
12. 参考来源
```

---

## Step 10：输出可复制提示词

最后必须输出一个提示词，让用户下次能复用。

提示词应包含：

* 角色；
* 任务；
* 参考文档写法；
* 输出结构；
* 写作要求；
* 表格规则；
* 风格要求；
* 禁止事项。

---


# 六、详细输出模板

## 1. 文档定位判断

输出：

```text
这份文档属于【文档类型】。
它面向【目标读者】。
它的核心目的不是【错误理解】，而是【真实目的】。
它的使用场景是【汇报/方案/立项/研究/复盘】。
```

示例：

```text
这份文档属于内部深度研究报告，面向产品、方案、技术和管理层。它的核心目的不是介绍 MaaS 安全概念，而是形成一套可用于平台建设和方案设计的安全能力框架。
```

---

## 2. 一句话主线

输出：

```text
这份文档的主线是：
【一句话概括】
```

示例：

```text
这份文档的主线是：随着平台从模型 API 走向 RAG、工具调用和 Agent 执行，安全边界持续外扩，因此安全能力必须从接口防护升级为可信执行架构。
```

---

## 3. 整体结构拆解

输出表格：

| 模块   | 内容         | 实际作用    |
| ---- | ---------- | ------- |
| 封面   | 研究定位、对象、范围 | 定义报告边界  |
| 执行摘要 | 核心判断、推荐架构  | 先给领导结论  |
| 第一章  | 问题本质       | 解释为什么研究 |
| 第二章  | 政策标准       | 补充外部约束  |
| 第三章  | 架构范式       | 归纳行业做法  |
| 第四章  | 平台案例       | 验证行业落地  |
| 第五章  | 推荐架构       | 提出自己的框架 |
| 后续章节 | 专题、风险、路线   | 转向落地执行  |

---

## 4. 章节推进逻辑

输出：

```text
这份文档的章节推进不是平铺的，而是递进式的：

第一步：先定义研究对象和问题本质；
第二步：补充政策、标准、行业约束；
第三步：归纳主流架构范式；
第四步：用平台案例验证这些范式；
第五步：提出自己的推荐架构；
第六步：围绕推荐架构展开专题；
第七步：落到风险控制和路线图。
```

---

## 5. 段落写法规律

输出：

```text
这份文档的段落通常遵循：
事实说明 → 原因解释 → 判断提升 → 对后文的引导
```

并给出可复用模板：

```text
【事实】当前……正在从……走向……
【原因】这一变化导致……
【判断】因此，核心问题不再是……而是……
【引导】后续应从……角度展开。
```

---

## 6. 表格插入规律

输出：

```text
这份文档中的表格主要出现在四类位置：
1. 复杂概念需要对比时；
2. 多阶段演进需要清晰呈现时；
3. 政策标准需要转成落地要求时；
4. 风险、控制措施和路线图需要执行化时。
```

并说明：

```text
表格通常不是先出现，而是在一段判断之后出现，用来支撑前面的结论。表格之后还会继续总结它对报告主线的启示。
```

```text
常用句式包括：
- 本报告认为……
- 核心问题不是……而是……

- 这种范式的优势是……弱点是……

```

## 8. 可复用写作框架

输出：

```text
如果要复刻这类文档，可以使用以下框架：

1. 研究定位
2. 执行摘要
3. 问题本质与演进

5. 主流范式分析

10. 建设路线图
11. 参考来源
```

---

## 9. 最值得学习的点

输出 5-8 条。

示例：

```text
1. 先定义问题本质，再进入资料分析。
2. 不把平台案例写成功能清单，而是抽象成架构范式。
3. 每章都有本章结论，读者不需要自己猜判断。
4. 表格用于支撑结构化判断，而不是堆信息。
5. 最后一定落到风险控制和建设路线，避免只停留在概念层。
```

---

## 10. 应避免的问题

输出：

```text
如果复刻这类文档，要避免：
1. 只总结内容，不提炼结构；
2. 只罗列资料，不形成主线；
3. 只列竞品功能，不归纳范式；
4. 只讲概念，不解释为什么出现；
5. 只写架构，不落到风险和路线；
6. 表格过多但没有前后解释；
7. 每章没有结论。
```

---

# 七、快速版指令

如果只想快速提炼，可以用这个短指令：

```text
请阅读我上传的文件，不要总结内容，而是提炼它的写作思路、结构逻辑和可复用方法。

请输出：
1. 文档定位；
2. 一句话主线；
3. 整体结构；
4. 章节推进逻辑；
5. 段落写法；
6. 表格/图示使用规则；
7. 语言风格；
8. 可复用写作框架；
9. 最值得学习的点；
10. 可直接复制的提示词。
```

# 八、深度版指令

如果想完整分析一个标杆文件，可以用这个：

```text
请把我上传的文件当成一个标杆样本，深入提炼它的写作思路、思考方式、结构设计和表达方法。

你的任务不是总结文档内容，而是拆解它为什么这样写。

请重点分析：
1. 这份文档的目标读者是谁；
2. 它服务什么决策或场景；
3. 它的核心主线是什么；
4. 它的目录为什么这样排；
5. 每个章节承担什么作用；
6. 每章开头和结尾如何处理；
7. 每个段落如何从事实走向判断；
8. 表格什么时候出现，为什么出现；
9. 图示/案例/截图如何服务论证；
10. 它如何提出自己的框架或结论；
11. 它如何避免资料堆砌；
12. 它的语言风格、判断句式、连接词是什么；
13. 这套写法如何迁移到其他主题；
14. 最后沉淀成一份可复用写作指令。

输出时请使用 Markdown，结构清晰，尽量具体。
```

# 九、模型执行时的检查清单

在执行这个指令时，模型必须自查：

| 检查项            | 是否必须 |
| -------------- | ---- |
| 是否区分了内容总结和写法提炼 | 必须   |
| 是否提炼了一句话主线     | 必须   |
| 是否解释了目录为什么这样排  | 必须   |
| 是否分析了每章作用      | 必须   |
| 是否提炼了段落结构      | 必须   |
| 是否分析了表格插入时机    | 必须   |
| 是否分析了语言风格      | 必须   |
| 是否给出可复用框架      | 必须   |
| 是否给出可复制提示词     | 必须   |
| 是否指出避免的问题      | 必须   |

# 十、最终口令

以后可以直接说：

```text
帮我按“写作思路提炼模式”分析这个文件：不要总结内容，重点提炼它的结构、主线、段落写法、表格使用规则、语言风格和可复用提示词。
```

收到这句话后，模型应默认按本指令执行。
