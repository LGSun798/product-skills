# 深度调研报告生成指令

## 0. 一句话定义

这是一套用于生成**深度调研报告**的完整指令，分为两部分：

1. **信息搜集指令**：用于指导模型围绕一个主题系统搜集资料、建立证据库、提炼研究主线。
2. **报告成文指令**：用于把搜集到的资料组织成一份结构完整、逻辑高级、可直接汇报的深度研究报告。

这套指令的目标不是让模型“多搜一点资料”，而是让模型完成：

```text
明确研究对象
→ 拆解研究问题
→ 搜集权威资料
→ 识别行业主线
→ 归纳架构/范式
→ 对比典型案例
→ 抽象推荐框架
→ 给出风险、路线图和落地建议
→ 输出可汇报的深度报告
```

---

# 第一部分：信息搜集指令

## 1. 信息搜集的核心目标

信息搜集阶段不是为了“找很多链接”，而是为了回答 6 个问题：

1. 这个领域/概念为什么出现？
2. 它解决了什么旧问题？
3. 它现在发展到什么阶段？
4. 主流玩家、政策、标准、架构、技术路线分别是什么？
5. 它真正落地时有哪些风险、难点和控制措施？
6. 最终能不能抽象成一个可用于项目方案或产品设计的框架？

搜集资料时，必须避免：

* 只搜定义；
* 只找新闻；
* 只列竞品；
* 只堆政策；
* 只做功能表格；
* 只写概念解释；
* 没有研究主线；
* 没有落地判断。

---

## 2. 信息搜集前必须先确认的问题

在开始搜集前，先确认以下信息。

如果用户没有提供，模型应主动追问；如果时间紧，可以自行做合理假设并在报告中说明。

| 问题                 | 作用                   |
| ------------------ | -------------------- |
| 研究主题是什么            | 明确调研对象               |
| 报告给谁看              | 决定深度、语言和侧重点          |
| 报告目的是什么            | 决定是科普、立项、方案、竞品还是战略判断 |
| 是否需要最新信息           | 决定是否必须联网搜索           |
| 是否偏技术、产品、商业、政策还是落地 | 决定资料搜集方向             |
| 是否要形成推荐方案          | 决定是否需要抽象架构和路线图       |
| 是否有公司/项目背景         | 决定结论是否要结合实际业务        |
| 是否有参考报告风格          | 决定结构和排版              |

---

## 3. 信息搜集的主问题拆解

对任何深度调研主题，都先拆成 10 类问题。

## 3.1 概念与边界

要搜集：

* 这个概念的一句话定义；
* 官方定义；
* 小白理解版定义；
* 它的上位概念；
* 它的同类概念；
* 它的下位模块；
* 容易混淆的概念；
* 它不包括什么。

目标：

> 明确研究对象，防止报告边界漂移。

---

## 3.2 背景与问题本质

要搜集：

* 它为什么出现；
* 原来没有它时行业怎么做；
* 旧方案有什么痛点；
* 新技术/新政策/新业务为什么推动它出现；
* 它解决的是效率问题、安全问题、成本问题、合规问题，还是商业模式问题；
* 这个问题为什么现在变重要。

目标：

> 找到整份报告的“问题本质”。

深度报告必须有这一层，否则只是资料整理。

---

## 3.3 行业演进阶段

要搜集：

* 早期阶段是什么；
* 当前阶段是什么；
* 下一阶段可能是什么；
* 每个阶段的平台形态、核心问题、代表能力；
* 哪些能力已经成熟；
* 哪些能力正在落地；
* 哪些能力还在探索。

推荐输出表：

| 阶段 | 平台/产品形态 | 核心问题 | 代表能力 | 成熟度 |
| -- | ------- | ---- | ---- | --- |

目标：

> 让报告具备时间纵深，而不是只写当前状态。

---

## 3.4 政策、标准与监管要求

如果主题涉及 AI、安全、医疗、金融、数据、教育、政企等，一定要搜集政策和标准。

要搜集：

* 国内政策；
* 国外政策；
* 国家标准；
* 行业标准；
* 国际标准；
* 监管要求；
* 合规框架；
* 标准对产品能力的影响；
* 标准要求如何映射到功能模块或安全域。

推荐输出表：

| 政策/标准 | 时间/状态 | 核心要求 | 对产品/平台的影响 |
| ----- | ----- | ---- | --------- |

目标：

> 不只写政策名称，而是把政策转成产品能力、控制项、交付要求。

---

## 3.5 主流架构范式 / 产品范式

深度报告必须归纳“范式”。

要搜集：

* 行业内有哪些主流路线；
* 每种路线解决什么问题；
* 每种路线的核心组件；
* 每种路线的优点；
* 每种路线的不足；
* 哪些路线适合当前项目；
* 是否可以组合成推荐架构。

推荐输出表：

| 架构/产品范式 | 核心控制点 | 适合解决的问题 | 不足 | 对本项目启示 |
| ------- | ----- | ------- | -- | ------ |

目标：

> 从“谁有什么功能”上升到“行业有几种做法”。

---

## 3.6 典型平台 / 竞品 / 案例

要搜集：

* 国内平台；
* 国外平台；
* 开源项目；
* 标杆产品；
* 行业案例；
* 大厂方案；
* 官方白皮书；
* 产品文档；
* 客户案例。

但不要只列功能。

每个案例要回答：

1. 它的定位是什么？
2. 它解决什么问题？
3. 它的架构思路是什么？
4. 它有哪些可借鉴能力？
5. 它有什么不足？
6. 对当前项目有什么启示？

推荐输出表：

| 平台/案例 | 定位 | 核心能力 | 架构特点 | 可借鉴点 | 不足 |
| ----- | -- | ---- | ---- | ---- | -- |

目标：

> 从案例中提炼共性能力，而不是做竞品功能清单。

---

## 3.7 学术研究与新兴概念

适用于技术型、AI 型、安全型、科研型报告。

要搜集：

* 最新论文；
* 学术热点；
* 新兴术语；
* 开源框架；
* 技术社区讨论；
* 安全攻击/防御方法；
* 仍未成熟的问题。

每个新概念都要回答：

1. 它为什么出现？
2. 它解决什么旧问题？
3. 它和已有概念有什么区别？
4. 当前成熟度如何？
5. 对产品落地有什么价值？

推荐输出表：

| 新概念 | 出现原因 | 解决问题 | 成熟度 | 落地价值 |
| --- | ---- | ---- | --- | ---- |

目标：

> 把新概念讲成问题演进，而不是术语堆叠。

---

## 3.8 风险模型与控制清单

要搜集：

* 主要风险；
* 攻击链路；
* 失败场景；
* 控制措施；
* 风险等级；
* 检查清单；
* 成熟度模型；
* 上线前检查项。

推荐输出表：

| 风险点 | 触发场景 | 影响 | 控制措施 | 优先级 |
| --- | ---- | -- | ---- | --- |

目标：

> 让报告从“理解行业”进入“可落地治理”。

---

## 3.9 商业价值与落地路线

要搜集：

* 谁使用；
* 谁买单；
* 谁推动；
* 谁反对；
* 成本在哪里；
* 价值在哪里；
* 短期能做什么；
* 中期能做什么；
* 长期能做什么；
* MVP 版本应该包含什么；
* 哪些能力后置。

推荐输出表：

| 阶段 | 时间 | 目标 | 核心能力 | 产物 | 商业价值 |
| -- | -- | -- | ---- | -- | ---- |

目标：

> 让报告从“研究”落到“行动”。

---

## 3.10 参考来源与证据库

要搜集：

* 官方文档；
* 政策原文；
* 标准原文；
* 平台产品文档；
* 白皮书；
* 学术论文；
* 权威机构报告；
* 开源项目；
* 可信媒体报道。

每条资料要记录：

| 来源 | 类型 | 可信度 | 关键内容 | 可支撑章节 |
| -- | -- | --- | ---- | ----- |

目标：

> 保证报告有事实支撑，不是模型编出来的。

# 4. 信息来源优先级

搜集资料时，按以下优先级使用来源。

## 4.1 第一优先级：权威原始来源

优先使用：

* 政府官网；
* 标准组织；
* 官方产品文档；
* 官方白皮书；
* 企业安全/技术文档；
* 论文原文；
* 开源项目官方仓库；
* 法规原文。

这些可以作为核心依据。

---

## 4.2 第二优先级：行业研究与专业机构

可以使用：

* Gartner；
* Forrester；
* IDC；
* NIST；
* OWASP；
* ISO；
* CNCF；
* Linux Foundation；
* 中国信通院；
* TC260；
* 行业协会；
* 安全公司报告。

这些适合支撑趋势、框架和行业判断。

---

## 4.3 第三优先级：媒体、博客、社区

可以使用，但只能作为补充：

* 技术博客；
* 企业公众号；
* 新闻报道；
* Medium；
* Reddit；
* Hacker News；
* 知乎；
* 公众号文章；
* 个人博客。

这类资料不能作为关键结论的唯一依据。

对每个主题生成 5 类关键词。

* 相关政策词；

* “架构”；

# 6. 信息搜集输出格式

搜集阶段不要直接写报告，先输出“调研素材库”。

推荐结构：

```text id="qyxy5j"
1. 研究对象与边界
2. 初步核心判断
3. 资料来源总览
4. 政策与标准资料
5. 行业范式资料
6. 平台案例资料
7. 技术/学术资料
8. 风险与控制资料
9. 路线图与落地资料
10. 待确认问题
```

每个资料点要写：

```text id="e52e0a"
来源：
类型：
可信度：
核心内容：
可支撑报告章节：
是否需要二次验证：
```

# 7. 信息搜集充分性的判断标准

只有满足以下条件，才可以进入成文阶段。

| 检查项   | 标准              |
| ----- | --------------- |
| 定义清楚  | 已明确研究对象和边界      |
| 背景清楚  | 已解释为什么出现、解决什么问题 |
| 政策/标准 | 至少覆盖主要政策或权威框架   |
| 行业范式  | 至少归纳 3 种以上主流路线  |
| 平台案例  | 至少覆盖国内外典型案例     |
| 风险清单  | 已识别主要风险和控制措施    |
| 落地路线  | 已形成短中长期建议       |
| 来源质量  | 核心结论有权威来源支撑     |
| 待验证项  | 未确认内容已明确标注      |

如果不满足，不要直接写报告，应继续搜集或向用户提问。

# 8. 信息搜集指令模板

## 8.1 单独用于搜集资料

```text id="j64zcm"
请围绕【主题】做一轮深度调研，先不要写完整报告。

调研目标：
我要判断【主题】的行业现状、关键问题、主流架构/方案、典型案例、风险控制和落地路线。

请按以下方向搜集信息：
1. 概念定义与边界：它是什么，不是什么，上位/同类/下位概念是什么。
2. 背景与问题本质：它为什么出现，解决了什么旧问题，为什么现在重要。
3. 行业演进阶段：从早期到当前再到下一阶段，分别是什么形态、问题和能力。
4. 政策、标准与监管要求：国内外主要政策、标准、框架，以及它们对产品/平台能力的要求。
5. 主流架构范式/产品范式：行业里有哪些主流路线，各自解决什么问题，有什么不足。
6. 典型平台/竞品/案例：国内外代表平台、官方架构、能力特点、可借鉴点。
7. 学术研究与新兴概念：最新论文、开源项目、新兴术语、研究热点、成熟度。
8. 风险模型与控制清单：主要风险、攻击/失败链路、控制措施、上线检查项。
9. 商业价值与落地路线：用户角色、买单方、落地路径、短中长期路线图。
10. 参考来源：列出所有关键来源，区分官方、标准、论文、平台文档、媒体。

输出要求：
- 不要直接成文报告，先输出调研素材库。
- 每条资料都要说明来源、可信度、核心内容、可支撑报告的哪一章。
- 对未验证内容明确标注“待验证”。
- 最后给出：初步核心判断、资料缺口、下一步需要补充的问题。
```

---

## 8.2 用于快速搜集 + 初步判断

```text id="vjfp6f"
请围绕【主题】做快速深度调研，并输出初步判断。

输出格式：

- 行业演进主线

```

# 第二部分：深度调研报告成文指令

## 9. 深度报告的核心主线

深度调研报告不是“百科解释”，而是要形成一条研究主线。

推荐主线：

```text id="l3wu9p"
问题为什么出现
→ 行业如何演进
→ 政策/标准如何约束
→ 主流方案有哪些
→ 标杆平台怎么做
→ 我们如何抽象自己的框架
→ 关键风险如何控制
→ 最后如何落地
```

报告必须有自己的判断，而不是复述资料。

# 10. 深度报告标准结构

推荐结构如下：

```text id="otzstx"
封面
目录
执行摘要
第一章 问题本质与行业演进
第二章 政策、标准与监管要求
第三章 主流架构范式 / 产品范式分析
第四章 典型平台与案例研究
第五章 推荐框架 / 推荐架构
第六章 核心能力专题一
第七章 核心能力专题二
第八章 学术热点与新兴概念
第九章 风险模型与控制清单
第十章 建设路线图与落地建议
参考来源
附录
```

不同主题可以调整章节名称，但逻辑不能乱。

# 11. 封面怎么写

封面必须体现研究定位。

## 11.1 封面内容

| 内容   | 写法                        |
| ---- | ------------------------- |
| 报告标题 | XX 能力演进与落地研究报告            |
| 副标题  | 从 A 到 B / 面向 XX 场景的能力框架   |
| 报告定位 | 内部研究报告 / 方案设计参考 / 项目研判材料  |
| 报告日期 | 年月                        |
| 适用对象 | 产品、研发、安全、销售、咨询、管理层        |
| 研究范围 | 覆盖哪些对象，不覆盖哪些对象            |
| 核心方法 | 政策研判、平台案例、架构分析、风险建模、路线图设计 |

## 11.2 封面示例

```text id="3lwglx"
XX 平台能力演进与落地研究报告
从【旧范式】到【新范式】

研究定位：
本报告面向【项目/产品/行业】建设、咨询和评估场景，围绕【问题本质】、【政策标准】、【平台案例】、【推荐架构】、【风险控制】和【建设路线】展开，目标是形成可用于方案设计和项目推进的能力框架。
```

# 13. 执行摘要怎么写

执行摘要是深度报告的灵魂。

必须包含：

1. 核心判断；
2. 行业演进主线；
3. 推荐框架；
4. 落地优先级；
5. 关键风险；
6. 建设建议。

## 13.1 执行摘要结构

```text id="a5eptq"
核心判断
架构/框架补强判断
行业演进主线表
可落地优先级
后续章节如何展开
```

## 13.2 执行摘要写法

示例：

```text id="gy0vyv"
本报告认为，【主题】的演进不是传统能力的简单叠加，而是随着【业务/技术/场景】扩展，问题边界持续外移。下一阶段的核心问题不是单纯解决【旧问题】，而是治理由【主体 A、主体 B、主体 C】共同组成的新型系统。

因此，【主题】能力应从【旧范式】升级为【新范式】。建议采用本报告提出的【推荐框架】，以【核心对象】为起点，以【核心资源】为中心，以【关键控制】为中枢，以【执行/落地对象】为边界，以【持续治理】为闭环。
```

# 14. 第一章：问题本质与行业演进

## 14.1 章节作用

说明这个主题为什么重要，为什么现在必须研究。

## 14.2 必须回答

* 原来是什么形态；
* 现在变成什么形态；
* 为什么安全/产品/业务边界发生变化；
* 新问题从哪里来；
* 旧方法为什么不够；
* 未来会走向哪里。

## 14.3 推荐结构

```text id="89qd84"
1.1 从旧形态走向新形态
1.2 问题边界为什么外扩
1.3 新概念出现的根本原因
1.4 企业级落地链路
1.5 关键用户角色与痛点
```

## 14.4 写法要求

每个新概念都必须解释“为什么出现”。

不要写：

> Prompt Injection 是一种攻击方式。

要写：

> Prompt Injection 出现的根本原因，是自然语言输入同时具备数据属性和指令属性，传统输入校验无法判断用户是在提问还是在试图改写模型行为。

# 15. 第二章：政策、标准与监管要求

## 15.1 章节作用

把政策标准转成产品/平台能力要求。

## 15.2 必须搜集

* 国内政策；
* 国外政策；
* 国家标准；
* 国际标准；
* 行业框架；
* 安全规范；
* 合规要求；
* 监管趋势。

## 15.3 推荐结构

```text id="1u1u5h"
2.1 国内政策
2.2 国外框架
2.3 重点标准与主要内容
2.4 标准要求到能力域的映射
```

## 15.4 表格要求

必须有：

| 政策/标准 | 状态与时间 | 主要内容 | 落地含义 |
| ----- | ----- | ---- | ---- |

还应有：

| 安全域/能力域 | 标准依据 | 核心要求 | 报告后续对应章节 |
| ------- | ---- | ---- | -------- |

## 15.5 写法要求

不要只写政策名称。

必须写：

> 这个标准要求产品做什么，平台需要补什么能力，落地时要产出什么证据。

# 16. 第三章：主流架构范式 / 产品范式分析

## 16.1 章节作用

把行业做法从“平台功能”抽象成“范式”。

## 16.2 必须回答

* 行业内有几类主流路线；
* 每类路线的核心控制点是什么；
* 分别解决什么问题；
* 不足在哪里；
* 对本报告推荐框架有什么启示。

## 16.3 推荐结构

```text id="2b810o"
3.1 范式一
3.2 范式二
3.3 范式三
3.4 范式四
3.5 范式五
3.6 范式对比与组合判断
```

## 16.4 表格要求

| 架构范式 | 核心控制点 | 适合解决的问题 | 不足 | 对推荐架构的启示 |
| ---- | ----- | ------- | -- | -------- |

## 16.5 写法要求

这一章要有“本章结论”。

示例：

```text id="sd0y7q"
本章结论：
当前主流架构大体可归纳为 X 类。它们分别解决不同问题，没有单一范式能覆盖全部风险。企业级平台应采用组合架构：以 A 作为底座，以 B 作为控制中枢，以 C 作为执行控制点，以 D 形成闭环。
```

# 17. 第四章：典型平台与案例研究

## 17.1 章节作用

通过标杆平台案例，证明行业如何落地。

## 17.2 必须覆盖

* 国际平台；
* 国内平台；
* 开源项目；
* 头部厂商；
* 新兴玩家；
* 行业案例。

## 17.3 每个案例必须写

```text id="rjqv9h"
平台定位
核心能力
架构亮点
可借鉴点
不足或边界
对本报告的启示
```

## 17.4 禁止写法

不要写成：

> A 平台有功能 1、功能 2、功能 3。

要写成：

> A 平台体现的是某种架构范式，它说明行业正在把哪些能力平台化、哪些风险前置到运行时控制、哪些能力仍依赖客户自建。

## 17.5 表格要求

| 平台 | 架构重点 | 可借鉴能力 |
| -- | ---- | ----- |

还可以加：

| 维度 | 国外平台表现 | 国内平台表现 | 判断 |
| -- | ------ | ------ | -- |

# 18. 第五章：推荐框架 / 推荐架构

## 18.1 章节作用

这是深度报告的高级感来源。

不能只总结别人，要提出自己的推荐框架。

## 18.2 推荐框架写法

推荐使用：

```text id="4y0i4g"
X 平面、Y 层、Z 闭环
```

或：

```text id="g8341o"
三阶段、五能力、四闭环
```

或：

```text id="56zqx3"
一条主链路、三类控制点、六项基础能力
```

## 18.3 推荐架构必须回答

* 架构定位；
* 架构原则；
* 总体结构；
* 核心组件；
* 关键职责；
* 必须产出的证据；
* 如何支撑后续章节；
* 为什么比能力清单更好。

## 18.4 写法示例

```text id="83jp03"
本报告提出【框架名称】作为面向【场景】的推荐架构。该架构不是单一产品模块，也不是现有能力的简单叠加，而是围绕【核心对象】、【关键资源】、【执行链路】和【治理闭环】建立统一控制体系。
```

## 18.5 表格要求

| 平面/层级 | 核心组件 | 关键职责 | 必须产出的证据 |
| ----- | ---- | ---- | ------- |

以及：

| 闭环 | 目标 | 落地动作 |
| -- | -- | ---- |

# 20. 第八章：学术热点与新兴概念

## 20.2 每个概念必须回答

> A 是什么，B 是什么，C 是什么。

# 26. 报告成文指令模板

## 26.1 单独用于成文

```text id="tosjiw"

13. 未验证或资料不足的内容必须标注“待验证”。
```

以后时间紧，可以直接说：

```text id="h6j2os"
按我的深度调研报告指令来：先搜集政策标准、行业范式、平台案例、技术热点、风险控制和落地路线，再按“问题本质→政策标准→架构范式→平台案例→推荐框架→专题展开→风险控制→路线图”的结构输出报告。
```

---

# 29. 最终原则

深度调研报告的最高标准是：

```text id="m3vzzd"
不是资料更多，而是主线更清楚；
不是概念更多，而是问题本质更清楚；
不是案例更多，而是能抽象出范式；
不是表格更多，而是能支持判断；
不是写得更长，而是能指导下一步行动。
```

收到这类任务时，模型应默认按本指令执行。
